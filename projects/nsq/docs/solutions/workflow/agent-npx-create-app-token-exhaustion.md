# 문제
AI 에이전트 환경에서 `npx create-next-app` 또는 `npx create-react-app`과 같은 부트스트랩 명령어를 실행할 때, 순간적으로 엄청난 양의 입력/출력 토큰이 스트리밍되어 컨텍스트 윈도우(토큰 한도)를 초과하고 세션이 비정상 종료되는 현상이 발생함.

에러 메시지:
`Context window/Token limit exceeded` 또는 에이전트 응답 불능 및 행(Hang) 현상.

재현 조건:
- 에이전트 환경에서 명시적인 매개변수 설정 없이 대화식(Interactive) CLI 명령어(`npx create-next-app`)를 직접 실행할 때.
- 의존성 다운로드 및 설치 과정에서 실시간 진행률 스피너(Loader/Spinner) 및 프로그레스 바 스트리밍 출력을 제한 없이 화면에 노출시킬 때.

# 실패한 시도
- **일반적인 CLI 호출 시도**: `npx create-next-app@latest projects/nsq-shadowing`을 그대로 실행함.
  - *결과*: CLI 환경에서 "TypeScript를 사용할 것인가?" 등의 질문 프롬프트에서 멈추어(대기 상태) 입력 대기가 발생하고, 에이전트가 무한 루프에서 터미널을 읽어 토큰이 소진됨.
- **단순 백그라운드(&) 실행**: 프로세스만 백그라운드로 띄워 둠.
  - *결과*: 대화형 프롬프트가 여전히 활성화되어 프로세스가 대기하고, 설치 출력이 지속되어 원인을 규명하기 어려움.

# 해결책
1. **완전 비대화형 옵션(Non-interactive parameters) 부여**:
   모든 프롬프트 입력을 건너뛰기 위해 명령어 수준에서 옵션을 미리 명시하여 실행합니다.
   ```bash
   npx create-next-app@14 projects/nsq-shadowing --ts --tailwind --eslint --app --src-dir --import-alias "@/*" --use-npm --yes
   ```
2. **출력 스트리밍 억제 및 무음 실행 (Silent/Log suppression)**:
   스피너나 진척도가 화면에 출력되지 않도록 `--silent` 플래그를 활용하거나, 출력을 `/dev/null`로 리다이렉트합니다.
   ```bash
   npm install --silent > /dev/null 2>&1
   ```
3. **비동기 처리 시간(WaitMsBeforeAsync) 최소화 및 백그라운드 태스크 활용**:
   설치가 완료될 때까지 에이전트가 동기적으로(Synchronously) 대기하면서 실시간 터미널 덤프를 계속 읽지 않도록, 명령어를 백그라운드로 전송하고 완료 시 알림을 받게 합니다.

# 원인 분석
1. **CLI 대화형 입력의 한계**:
   샌드박스 터미널 환경에서 대화형 프롬프트는 입력을 강제하지만, 에이전트는 적절한 타이밍에 입력을 주지 못하면 지속적으로 터미널의 출력을 읽어오는 폴링 상태를 유지하여 컨텍스트 토큰을 고갈시킵니다.
2. **로딩 애니메이션의 프레임 업데이트**:
   터미널에서 1초에 여러 번 지워지고 그려지는 스피너(Spinner) 문자는 에이전트 스트림 상에서 개별적인 텍스트 덤프로 변환되어 중복 입력됩니다. 이로 인해 수십만 토큰에 이르는 방대한 데이터가 순식간에 컨텍스트 이력에 누적되어 토큰 한도를 넘어섭니다.

# 재발 방지
- 새로운 프레임워크나 보일러플레이트 설치 스킬 작성 시, 반드시 매개변수를 생략 없이 입력하는 단일 비대화형 명령어로 구성합니다.
- 패키지 매니저 실행 명령은 가능하면 무음(`--silent` 등) 및 출력 리다이렉션(`> /dev/null 2>&1`) 옵션을 기본값으로 사용합니다.
- 장시간 실행되는 설치 명령어는 동기 대기 시간을 짧게 설정하고, 백그라운드 비동기 태스크로 즉시 이관하여 진행 상황에 따른 토큰 유입을 방지합니다.
