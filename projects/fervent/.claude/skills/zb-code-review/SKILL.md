# zb-code-review

리뷰를 **요청할 때**와 **받을 때** 두 가지 워크플로우를 다룬다. 감사 표현 금지, 기술적 검증 우선.

---

## Part 1 — 리뷰 요청 (Requesting Code Review)

구현이 완료되면 서브에이전트에게 코드를 검토시킨다. 서브에이전트는 현재 세션 히스토리 없이 코드 산출물만 평가한다.

### 언제 요청하는가

**필수:**
- 서브에이전트 기반 개발(subagent-driven development) 각 태스크 완료 후
- 주요 기능 구현 완료 후
- main 브랜치 병합 전

**선택:**
- 막혔을 때 (새로운 시각)
- 복잡한 버그 수정 후

### 요청 절차

**1. Git SHA 확인:**
```bash
BASE_SHA=$(git rev-parse HEAD~1)  # 또는 origin/main
HEAD_SHA=$(git rev-parse HEAD)
```

**2. 서브에이전트 디스패치:**

`Task` 도구 (`general-purpose` 타입)를 사용하여 `subagent-reviewer-prompt.md` 템플릿을 채워 전달한다.

**플레이스홀더:**
- `{DESCRIPTION}` — 구현 내용 한 줄 요약
- `{PLAN_OR_REQUIREMENTS}` — 요구사항 또는 플랜 파일 경로
- `{BASE_SHA}` — 시작 커밋
- `{HEAD_SHA}` — 종료 커밋

**3. 피드백 처리:**
- Critical → 즉시 수정
- Important → 다음 단계 진행 전 수정
- Minor → 메모 후 나중에 처리
- 틀린 피드백 → 기술적 근거로 반박

## Part 2 — 리뷰 피드백 수신 (Receiving Code Review)

피드백은 감정적 반응이 아니라 기술적 평가의 대상이다.

**핵심 원칙:** 검증 후 구현. 가정 전에 질문. 사회적 편의보다 기술적 정확성.

### 반응 패턴

```
피드백 수신 시:

1. 읽기: 반응 없이 전체 피드백 완독
2. 이해: 요구사항을 자신의 말로 재구성 (불명확하면 질문)
3. 검증: 코드베이스 현실과 대조 확인
4. 평가: 이 코드베이스에 기술적으로 타당한가?
5. 응답: 기술적 인정 또는 근거 있는 반박
6. 구현: 항목별 하나씩, 각각 테스트
```

### 금지 응답

**절대 하지 말 것:**
- "맞습니다!" / "좋은 지적이에요!" (성과 과시형)
- "바로 구현할게요!" (검증 전 선언)
- "감사합니다" / "덕분에" / 어떤 형태의 감사 표현도 금지

**대신:**
- 기술적 요구사항을 재진술
- 불명확한 부분은 질문
- 틀린 경우 기술적 근거로 반박
- 그냥 수정 시작 (행동 > 말)

### 불명확한 피드백 처리

```
항목 중 하나라도 불명확하면:
  즉시 멈춤 — 어떤 항목도 구현하지 않는다
  불명확한 항목에 대해 명확화 요청

이유: 항목들이 연관될 수 있다. 부분적 이해 = 잘못된 구현.
```

**예시:**
```
리뷰어: "1번~6번 수정"
1,2,3,6번은 이해됨. 4,5번 불명확.

❌ 틀림: 1,2,3,6번 먼저 구현, 나중에 4,5번 질문
✅ 맞음: "1,2,3,6번은 이해했습니다. 4번과 5번은 진행 전 명확화가 필요합니다."
```

### 피드백 출처별 처리

**파트너(사용자)로부터:**
- 신뢰하되 범위가 불명확하면 질문
- 기술적 인정 또는 바로 실행

**외부 리뷰어로부터:**
```
구현 전 확인:
  1. 이 코드베이스에 기술적으로 정확한가?
  2. 기존 기능을 망가뜨리는가?
  3. 현재 구현에 이유가 있는가?
  4. 모든 플랫폼/버전에서 동작하는가?
  5. 리뷰어가 전체 컨텍스트를 파악하고 있는가?

제안이 틀린 것 같으면:
  기술적 근거로 반박

검증이 어려우면:
  "X 없이는 검증 불가. [조사/질문/진행] 중 어떻게 할까요?"

파트너의 기존 결정과 충돌하면:
  파트너와 먼저 논의
```

### YAGNI 체크

```
리뷰어가 "제대로 구현"을 제안하면:
  코드베이스에서 실제 사용 여부 확인

  미사용이면: "이 엔드포인트가 호출되지 않습니다. 제거하는 게 낫지 않을까요? (YAGNI)"
  사용 중이면: 제대로 구현
```

### 구현 순서

```
복수 항목 피드백의 경우:
  1. 불명확한 항목 먼저 명확화
  2. 아래 순서로 구현:
     - 차단 이슈 (기능 파괴, 보안)
     - 단순 수정 (오타, 임포트)
     - 복잡한 수정 (리팩토링, 로직)
  3. 항목별 개별 테스트
  4. 회귀 없음 확인
```

### 반박 기준

반박해야 하는 경우:
- 기존 기능을 망가뜨리는 제안
- 리뷰어가 전체 컨텍스트를 모르는 경우
- YAGNI 위반 (미사용 기능)
- 이 스택에 기술적으로 부정확
- 레거시/호환성 이유 존재
- 파트너의 아키텍처 결정과 충돌

**반박 방법:**
- 기술적 근거 사용, 방어적 태도 금지
- 구체적 질문
- 동작하는 테스트/코드 참조
- 아키텍처 이슈면 파트너 포함

### 올바른 피드백 인정 시

피드백이 맞을 때:
```
✅ "수정했습니다. [변경 내용 한 줄]"
✅ "정확합니다 — [구체적 이슈]. [위치]에서 수정."
✅ [그냥 수정하고 코드로 보여주기]

❌ "맞습니다!"
❌ "좋은 지적이에요!"
❌ "감사합니다!"
❌ 어떤 감사 표현도 금지
```

**왜 감사 금지:** 행동이 말한다. 그냥 수정하라. 코드 자체가 피드백을 들었음을 증명한다.

### 내 반박이 틀렸을 때

```
✅ "맞았습니다 — [X]를 확인했고 실제로 [Y]입니다. 지금 구현합니다."
✅ "검증했고 맞습니다. 초기 이해가 [이유]로 틀렸습니다. 수정 중."

❌ 긴 사과
❌ 반박했던 이유 방어
❌ 과도한 설명
```

사실로 정정하고 계속 진행.

### 흔한 실수

| 실수 | 수정 |
|------|------|
| 과시형 동의 | 요구사항 재진술 또는 바로 실행 |
| 맹목적 구현 | 먼저 코드베이스 대조 검증 |
| 일괄 처리 후 테스트 | 항목별, 각각 테스트 |
| 리뷰어가 맞다고 가정 | 기능 파괴 여부 확인 |
| 반박 회피 | 기술적 정확성 > 편안함 |
| 부분 구현 | 전체 항목 명확화 먼저 |
| 검증 불가인데 진행 | 한계 말하고 방향 요청 |

## GitHub 인라인 댓글 답변

인라인 리뷰 댓글에 답변할 때는 최상위 PR 댓글이 아닌 해당 댓글 스레드에 답변한다.

```bash
gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies \
  -f body="[답변 내용]"
```

## 핵심 원칙

- **외부 피드백 = 평가할 제안이지 따를 명령이 아니다**
- 검증하라. 의심하라. 그 다음 구현하라
- 과시형 동의 금지. 기술적 엄밀성 항상 유지
