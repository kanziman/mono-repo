# 코드 리뷰 오케스트레이터

**실행 모드:** 서브 에이전트 (Sub-agents) — 전문가는 필요한 순간에만 온디맨드로 호출되며, 결과는 오케스트레이터에게 반환된다.

**아키텍처 패턴:** 전문가 풀 (Expert Pool) — 요청 유형에 따라 적합한 전문가 에이전트를 선택적으로 호출한다.

---

## Phase 0: 컨텍스트 확인

실행 시작 전 기존 작업 맥락을 파악한다.

- `_workspace/` 폴더가 존재하고 사용자가 "다시", "재실행", "업데이트", "이전 결과 기반으로" 표현을 사용했다면 → **부분 재실행** (해당 전문가만 재호출)
- 그 외 → **초기 실행**

## Phase 1: 리뷰 범위 결정

요청을 분석하여 어떤 전문가를 호출할지 결정한다.

| 요청 유형 | 호출할 전문가 |
|---|---|
| 종합 리뷰 / PR 리뷰 / 전체 점검 / 요청이 모호한 경우 | security-reviewer + linter-agent + perf-analyst |
| 보안, 취약점, 인증/인가, injection | security-reviewer |
| 린트, 스타일, 타입 오류, 네이밍 | linter-agent |
| 성능, 병목, 메모리, 느린 쿼리 | perf-analyst |
| DB 마이그레이션 작성/검토 | db-migration 스킬 직접 참조 |

## Phase 2: 전문가 에이전트 호출

결정된 전문가를 서브 에이전트로 호출한다. 독립적인 전문가 여러 명은 `run_in_background: true`로 병렬 실행한다.

각 에이전트에 전달할 컨텍스트:
- 검토 대상 파일 경로 (또는 diff 내용)
- 사용자의 원래 요청
- 프로젝트 컨텍스트 (TypeScript ES modules, CLAUDE.md 규칙)

결과는 `_workspace/{전문가명}_results.md`에 저장하도록 에이전트에 지시한다.

## Phase 3: 결과 종합

각 전문가의 결과를 통합하여 아래 형식으로 보고한다.

```
## 코드 리뷰 결과

### 🔴 즉시 수정 필요 (Critical)
...

### 🟡 개선 권장 (Warning)
...

### 🟢 정보성 메모 (Info)
...

### ✅ 이상 없음
...
```

전문가 간 상충되는 의견은 삭제하지 않고 출처를 함께 표기한다.

## 에러 핸들링

- 특정 전문가 에이전트 실패 시 1회 재시도
- 재시도 후에도 실패하면 해당 전문가 결과 누락을 명시하고 나머지 결과로 계속 진행
- 검토 대상 파일이 없거나 접근 불가 시 사용자에게 경로 확인 요청

## 테스트 시나리오

**정상 흐름:** "src/auth/login.ts 코드 리뷰해줘" → Phase 1에서 종합 리뷰 판정 → 3 전문가 병렬 호출 → 통합 보고서 출력

**에러 흐름:** linter-agent 실패 → 1회 재시도 → 재실패 시 "linter-agent 결과 누락" 명시 후 security + perf 결과로 보고서 작성
