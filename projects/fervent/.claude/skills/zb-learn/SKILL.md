# zb-learn

문제를 해결한 직후 지식을 `docs/solutions/`에 구조화 문서로 저장한다.
컨텍스트가 가장 신선한 지금이 문서화 적기다.

**시작 선언:** `"I'm using the zb-learn skill to document this solution."`

---

## 🛠️ 핵심 체크리스트

1. `[ ]` **시작 선언**
2. `[ ]` **문제/해결책 추출**: 대화 히스토리에서 문제, 실패한 시도, 해결책을 파악한다
3. `[ ]` **분류**: 카테고리 결정 → 파일명(slug) 생성
4. `[ ]` **중복 확인**: `docs/solutions/`에 동일 문제 문서가 이미 있는지 간단히 확인 (grep)
5. `[ ]` **문서 작성**: 디렉토리 없으면 생성 후 5섹션 형식으로 작성
6. `[ ]` **완료 보고**

---

## 📂 저장 경로

```
docs/solutions/[category]/[slug].md
```

**카테고리 (자동 감지):**

| 카테고리 | 해당 문제 유형 |
|----------|----------------|
| `bug-fixes/` | 버그, 에러, 런타임 문제, 테스트 실패 |
| `architecture/` | 구조적 결정, 설계 패턴, 시스템 구성 |
| `workflow/` | 개발 프로세스, 도구 설정, CI/CD |
| `best-practices/` | 기타 모범 사례, 코딩 컨벤션 |

**slug 규칙:** `[문제-핵심-키워드].md` (영문 소문자, 하이픈 구분)
예: `npm-install-permission-error.md`, `react-useeffect-infinite-loop.md`

## 📝 문서 본문 구조 (5섹션)

### 1. 문제 (Problem)
증상, 에러 메시지, 재현 조건을 구체적으로 기술한다.

```
# 문제
[증상 설명]

에러 메시지:
[실제 에러 메시지]

재현 조건:
[어떤 상황에서 발생하는가]
```

### 2. 실패한 시도 (What Didn't Work)
시도했지만 실패한 접근들과 왜 실패했는지를 기록한다.

```
# 실패한 시도
- [시도 1]: [왜 안 됐는가]
- [시도 2]: [왜 안 됐는가]
```

### 3. 해결책 (Solution)
실제 동작하는 수정 내용을 코드 예시와 함께 기술한다.

````
# 해결책
[해결 방법 설명]

```[언어]
[코드 예시]
```
````

### 4. 원인 분석 (Why This Works)
왜 이 방법이 동작하는지 근본 원인을 설명한다.

```
# 원인 분석
[기술적 근본 원인 설명]
```

### 5. 재발 방지 (Prevention)
앞으로 같은 문제를 피하는 방법을 기술한다.

```
# 재발 방지
- [방지책 1]
- [방지책 2]
```

## 중복 확인 방법

```bash
# docs/solutions/ 안에서 핵심 키워드로 grep
grep -r "[핵심 키워드]" docs/solutions/ --include="*.md" -l
```

동일한 문제를 다루는 기존 문서가 있으면:
- **같은 문제, 같은 해결책** → 기존 문서에 현재 컨텍스트로 내용 보강 (`last_updated` 추가)
- **같은 문제, 다른 해결책** → 새 문서 생성 + 기존 문서 상호 링크

## 흔한 실수

| 실수 | 수정 |
|------|------|
| 해결책만 기록, 실패한 시도 누락 | 실패한 시도가 핵심 — 다음 사람이 같은 길을 걷지 않게 |
| 에러 메시지 생략 | 정확한 에러 메시지가 검색 가능성을 높임 |
| 중복 문서 생성 | 작성 전 grep으로 기존 문서 확인 |
| 추상적 설명 | 구체적 코드 예시와 재현 조건 포함 |
