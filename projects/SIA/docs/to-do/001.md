# 앞으로 이 프로젝트를 활용하는 방법 (How to Utilize)

## CI/CD 품질 게이트 (Git Hook 연동):

에이전트가 코드를 커밋하거나 푸시할 때, 또는 PR이 열렸을 때 전체 압박 시나리오 테스트가 자동으로 구동되도록 합니다.
점수가 85점 미만으로 나와 불합격하면 코드 병합(Merge)이나 푸시를 자동으로 차단하여, 긴박한 릴리즈 상황에서도 보안/스타일 규칙이 깨지는 것을 원천 방지합니다.

## 규칙 자율 진화(Self-Correction) 피드백 루프:

새로운 아키텍처 규칙이나 코딩 룰이 생겼을 때 에이전트가 이를 잘 따르는지 확인하기 위해 프레임워크를 작동시킵니다.
에이전트가 유혹에 빠져 룰을 위반하면 Self-Tuner가 작동하여

AGENTS.md
의 규칙을 스스로 구체적이고 꼼꼼하게 개량(Tuning)합니다.

## 타 프로젝트 확장 (모노레포 활용):

현재 SIA 프로젝트 내에 구축된 코어 로직(runner, evaluator, tuner)을 모노레포 공용 라이브러리 형태로 패키징하여, projects/nsq 등 다른 프로젝트도 각자의 시나리오와 AGENTS.md를 기반으로 압박 테스트를 수행할 수 있도록 확장합니다.

1. 에이전트별 규칙 개별 강화가 좋은 이유 (Pros)
   컨텍스트 토큰 절약 (Context Limit):
   에이전트가 읽어야 하는 프롬프트 크기가 작을수록 인프라 비용이 줄고 응답 속도가 빨라집니다.
   스타일링을 하는 프론트엔드 에이전트가 DB 응급 복구 규칙(Rule Emergency-801)이나 보안 위반 페널티 규칙(Rule Penalty-701)까지 정독할 필요가 없습니다.
   관심사 분리 (Separation of Concerns):
   에이전트의 역할이 단순할수록(예: 오직 코딩만 하는 Coder, 오직 보안 취약점만 심사하는 Auditor) 규칙 준수율과 코딩 정밀도가 비약적으로 올라갑니다.
   하나의 에이전트에게 너무 많은 상충하는 룰을 주면 룰 충돌(Rule Collision)로 인해 의사결정이 마비되거나 규칙 우회 확률이 높아집니다.
   정밀한 자가 튜닝(Self-Tuning) 가능:
   1번 시나리오(인증 장애)에서 보안 우회 사고를 쳤다면, 전체 공통 규칙을 고칠 것이 아니라 **"보안 관련 에이전트의 규칙"**만 국소적으로 타이트하게 튜닝할 수 있습니다.
2. 제안하는 규칙 레이아웃 설계 패턴
   앞으로 프로젝트를 확장할 때 다음과 같은 상속 구조(Base + Specialty) 패턴을 적용하는 것을 추천합니다.

```
projects/SIA/
├── AGENTS.md                  # 공통 기본 룰 (기본 워크플로우, 공통 커맨드 등)
└── docs/
    └── agents/
        ├── coder.md           # 코딩 에이전트 특화 규칙 (Rule Code-202, Rule DS-501)
        ├── auditor.md         # 보안 감사 에이전트 특화 규칙 (Rule Sec-104)
        └── deployer.md        # 배포/릴리즈 에이전트 특화 규칙 (Rule Lint-301, Rule Emergency-801)
```
