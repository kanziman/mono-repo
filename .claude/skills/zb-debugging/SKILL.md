# zb-debugging

이 스킬은 무작위 땜질식 임시 처방(Symptom Patching)을 철저히 금지하고, 장애의 **근본 원인(Root Cause)**을 파악하여 시스템 안정성 규격([projects/[프로젝트명]/docs/RELIABILITY.md](file:///projects/[프로젝트명]/docs/RELIABILITY.md))에 부합하는 영구적인 해결 방안을 적용하는 워크플로우를 정의합니다.

## 🚨 디버깅 철칙 (The Iron Law)

```text
근본 원인(Root Cause)의 입증 및 데이터 확인 전에는 어떠한 코드 수정도 Propose하거나 시도하지 마십시오.
```

---

## 🔄 체계적 디버깅 4단계 프로세스 (The 4 Phases)

### 1단계: 근본 원인 분석 (Root Cause Investigation)
*   **에러 분석**: 스택 트레이스 전체와 줄 번호, 에러 코드를 정독하십시오.
*   **재현성 확인**: 어떤 순서와 데이터 조건에서 오류가 발생하는지 재현 절차를 도출하십시오. (간헐적 오류 시 추측으로 때우지 말고 로그를 보강하여 데이터를 쌓으십시오.)
*   **역방향 추적 (Backward Tracing)**: 오류가 감지된 최종 지점에서 데이터가 생성된 최초 지점까지 함수 호출 체인을 역으로 따라가십시오 (Symptom이 아닌 Origin을 타격).
*   **다중 컴포넌트 경계 분석**: 마이크로서비스 또는 다층 레이어 구조인 경우, 각 경계면(API ➡️ DB, 빌드 스크립트 등)에 임시로 진단 로그를 삽입하여 어느 경계에서 전이가 왜곡되는지 규명하십시오.

### 2단계: 패턴 비교 분석 (Pattern Analysis)
*   **동작 사례 비교**: 동일한 코드베이스 내에서 정상적으로 잘 동작하는 유사 코드 패턴을 찾고, 고장 난 부분과의 차이점(차이가 아무리 작아도 무시 금지)을 비교하십시오.
*   **의존성 및 환경 검사**: 두 영역의 설정값(Environment), 설정 파라미터, 권한 등의 차이를 분석하십시오.

### 3단계: 가설 수립 및 최소 단위 테스트 (Hypothesis & Testing)
*   **단일 가설 수립**: "Y 상황 때문에 X가 오작동하는 것 같다"라는 구체적이고 단일한 가설을 설정합니다.
*   **최소단위 변수 통제**: 가설을 입증할 수 있는 가장 작고 단순한 코드 변화(한 번에 딱 하나의 변수만 수정)를 가해 작동 상태를 확인합니다.
    - 실패 시 추가 패치를 누적하지 말고 즉시 롤백한 뒤, 새로운 가설을 세우고 반복합니다.

### 4단계: 구현 및 신뢰성 검증 (Implementation & Verification)
*   **실패하는 테스트 추가**: 버그를 완벽히 재현하고 차후 회의적 회귀를 방지하기 위해 실패하는 단위/통합 테스트를 먼저 작성합니다 (`zb-TDD` 스킬 준수).
*   **신뢰성 규격 검사**: 네트워크, 외부 API 연동, 또는 시간 의존적인 버그를 수정할 때는 반드시 [projects/[프로젝트명]/docs/RELIABILITY.md](file:///projects/[프로젝트명]/docs/RELIABILITY.md)의 다음 사항을 충족하는지 검증하십시오:
    - 외부 호출 시 타임아웃(기본 5000ms)이 명시적으로 지정되었는가?
    - 일시적 오류 극복을 위해 지수 백오프(Exponential Backoff) 및 재시도 상한이 있는가?
    - 불필요한 부하 가중 방지를 위한 서킷 브레이커(Circuit Breaker)가 적절히 작동하는가?
*   **최종 성공 검증**: 수정한 후 전체 테스트 스위트가 온전히 통과하는지 확인합니다.

## 📚 지식 문서화 (Knowledge Capture)

4단계 완료(전체 테스트 통과) 후, 디버깅을 종료하기 전에 확인한다:

```
"문제가 해결됐습니다. 이 디버깅 과정을 zb-learn으로 문서화할까요?
 컨텍스트가 가장 신선한 지금이 적기입니다. (선택)"

- Yes → /zb-learn 실행
- No  → 디버깅 스킬 종료
```

이 질문은 non-blocking — 거절해도 디버깅 결과에 영향 없음.

## 🚨 아키텍처 의심 및 중단 규칙 (The 3-Fail Rule)

1.  **3번 이상 수정 실패 시 정지**: 한 에러에 대해 3번 이상 시도한 수정안이 실패했다면, 단순 버그가 아닌 아키텍처적 모순(Tight coupling, shared state 등)일 확률이 매우 높습니다.
2.  **즉시 정지 및 보고**: 4번째 임시 땜질을 중단하고 파트너(사용자)에게 아키텍처 설계를 근본적으로 다시 검토(Refactor)해야 함을 알리십시오.
3.  **기술 부채 기록**: 해당 설계상 결함을 즉시 [projects/[프로젝트명]/docs/exec-plans/tech-debt-tracker.md](file:///projects/[프로젝트명]/docs/exec-plans/tech-debt-tracker.md)에 정식 등록하여 기술 부채로 추적해야 합니다.
